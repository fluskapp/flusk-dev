# Flusk CLI - Usage Guide

## Quick Start

The Flusk CLI generates all code layers from entity schemas automatically.

### Installation

```bash
# Build the CLI
cd packages/cli
pnpm build
cd ../..

# Or use npm script
pnpm build:cli
```

### Basic Commands

```bash
# Generate from single entity
flusk g llm-call.entity.ts

# Generate from all entities
flusk g --all

# Generate types only (fast)
flusk g pattern.entity.ts --types-only

# Run migrations
flusk migrate
```

---

## Command Reference

### `flusk g [entity-file]`

Generate code from entity schema(s).

**Arguments:**
- `entity-file` - Entity file to generate from (e.g., `llm-call.entity.ts`)

**Options:**
- `--all` - Generate from all entities in packages/entities/src/
- `--types-only` - Generate only types package (skip resources, business-logic, execution)

**Examples:**

```bash
# Generate all layers for pattern entity
flusk g pattern.entity.ts

# Output:
# 📦 Processing: pattern.entity.ts
#   ✅ types/pattern.types.ts
#   ✅ resources/repositories/pattern.repository.ts
#   ✅ resources/migrations/009_patterns.sql
#   ✅ business-logic/pattern/validate-pattern.function.ts
#   ✅ execution/routes/pattern.routes.ts
#   ✅ execution/plugins/pattern.plugin.ts
#   ✅ execution/hooks/pattern.hooks.ts

# Generate from all entities
flusk g --all

# Generate only types (quick check)
flusk g conversion.entity.ts --types-only
```

## What Gets Generated

### 1. Types Package (`packages/types/`)

Generated for every entity:
- TypeScript types from TypeBox schemas
- JSON schemas for validation
- Insert/Update/Patch variants
- Proper imports and exports

**Example:** `llm-call.types.ts`
```typescript
/**
 * @generated by @flusk/cli from llm-call.entity.ts
 * DO NOT EDIT - regenerate using: flusk g llm-call.entity.ts
 */

export type LLMCallEntity = Static<typeof LLMCallEntitySchema>;
export type LLMCallInsert = Static<typeof LLMCallInsertSchema>;
export const LLMCallEntityJSONSchema = LLMCallEntitySchema;
```

### 2. Resources Package (`packages/resources/`)

**Repository:** Full CRUD operations
- `create()` - INSERT with validation
- `findById()` - SELECT by primary key
- `findMany()` - SELECT with pagination
- `update()` - UPDATE with validation
- `delete()` - DELETE with cascade

**Migration:** SQL schema
- Table creation with proper types
- Indexes for common queries
- Foreign key constraints
- Timestamps (created_at, updated_at)

**Example:** `llm-call.repository.ts` (split into multiple files <100 lines)

### 3. Business Logic Package (`packages/business-logic/`)

**Stubs only** - You implement the logic!

Generated stubs:
- `validate-{entity}.function.ts` - Validation stub
- `index.ts` - Barrel export

**Example:** `validate-pattern.function.ts`
```typescript
/**
 * @generated by @flusk/cli from pattern.entity.ts
 * DO NOT EDIT - regenerate using: flusk g pattern.entity.ts
 */

export interface ValidatePatternInput {
  pattern: unknown;
}

export interface ValidatePatternOutput {
  isValid: boolean;
  errors?: string[];
}

export function validatePattern(input: ValidatePatternInput): ValidatePatternOutput {
  // TODO: Implement validation logic
  return { isValid: true };
}
```

**You implement:** Custom business logic functions

### 4. Execution Package (`packages/execution/`)

**Routes:** REST API endpoints
- `POST /api/v1/{entities}` - Create
- `GET /api/v1/{entities}` - List with pagination
- `GET /api/v1/{entities}/:id` - Get by ID
- `PATCH /api/v1/{entities}/:id` - Update
- `DELETE /api/v1/{entities}/:id` - Delete

**Plugin:** Fastify plugin encapsulation
- Route registration
- Hook registration
- Schema validation
- Error handling

**Hooks:** Lifecycle hooks (composition layer)
- `beforeCreate` - Pre-insert logic
- `afterCreate` - Post-insert logic
- `beforeUpdate` - Pre-update logic
- `afterUpdate` - Post-update logic

## Regeneration Workflow

### When Entity Schema Changes

```bash
# 1. Edit entity schema (manual)
vim packages/entities/src/pattern.entity.ts

# 2. Regenerate all layers (automatic)
flusk g pattern.entity.ts

# 3. Your manual code is preserved:
#    ✅ Business logic functions (custom implementations)
#    ✅ Hooks (complex composition)
#    ✅ Middleware (auth, logging, etc.)
#
#    Updated automatically:
#    ✅ Types
#    ✅ Repositories
#    ✅ Migrations
#    ✅ Routes
#    ✅ Plugins
```

### Regenerate All Entities

```bash
# Use built-in script
pnpm regenerate

# Or manually
flusk g --all
```

## File Organization

### Generated Files (<100 lines each)

All generated files include header:
```typescript
/**
 * @generated by @flusk/cli from {entity}.entity.ts
 * DO NOT EDIT - regenerate using: flusk g {entity}.entity.ts
 */
```

### Split Repositories

Large repositories split into one function per file:
```
repositories/
└── pattern/
    ├── pool.ts           (database connection)
    ├── row-to-entity.ts  (data mapper)
    ├── create.ts         (INSERT)
    ├── find-by-id.ts     (SELECT by ID)
    ├── find-many.ts      (SELECT with pagination)
    ├── update.ts         (UPDATE)
    ├── delete.ts         (DELETE)
    ├── types.ts          (interfaces)
    └── index.ts          (barrel export)
```

Imports still work:
```typescript
import { create, findById } from '@flusk/resources/pattern.repository';
```

## CI/CD Integration

### Check for Manual Edits

```bash
# Validate no manual edits to @generated files
pnpm check:generated

# Exit code 0 = OK
# Exit code 1 = Manual edits detected
```

Add to CI pipeline:
```yaml
# .github/workflows/ci.yml
- name: Check generated files
  run: pnpm check:generated
```

### Auto-Regenerate on Schema Changes

```yaml
# .github/workflows/regenerate.yml
on:
  push:
    paths:
      - 'packages/entities/src/*.entity.ts'

jobs:
  regenerate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - run: pnpm install
      - run: pnpm regenerate
      - uses: stefanzweifel/git-auto-commit-action@v4
        with:
          commit_message: "chore: regenerate code from entities"
```

## Advanced Usage

### Custom Templates

Override default templates:
```bash
# Create custom template directory
mkdir -p .flusk/templates

# Copy default template
cp packages/cli/src/templates/repository.template.ts .flusk/templates/

# Edit template
vim .flusk/templates/repository.template.ts

# Generate with custom template
flusk g --template=.flusk/templates pattern.entity.ts
```

### Dry Run

Preview what would be generated:
```bash
flusk g pattern.entity.ts --dry-run

# Shows file paths and content preview
# Does not write files
```

### Force Overwrite

Overwrite existing files without prompt:
```bash
flusk g pattern.entity.ts --force

# Useful for CI/CD pipelines
```

## Troubleshooting

### CLI Not Found

```bash
# Build CLI first
cd packages/cli
pnpm build

# Or add to PATH
export PATH="$PATH:$(pwd)/packages/cli/dist/bin"
```

### Entity Not Found

```bash
# Error: Entity file not found: pattern.entity.ts

# Solution: Run from project root
cd /path/to/flusk
flusk g pattern.entity.ts

# Or use full path
flusk g packages/entities/src/pattern.entity.ts
```

### Import Errors After Generation

```bash
# Rebuild all packages
pnpm build

# Check imports in generated files
# They should use @flusk/* workspace aliases
```

### Generated Files Not Compiling

```bash
# Verify entity schema is valid TypeBox
# Run TypeScript compiler
pnpm build

# Check for syntax errors in entity
vim packages/entities/src/your-entity.entity.ts
```

## Best Practices

### 1. Entity Schema First
Always define entities first, then generate:
```bash
# 1. Create entity (manual)
vim packages/entities/src/new-entity.entity.ts

# 2. Generate all layers (automatic)
flusk g new-entity.entity.ts

# 3. Implement business logic (manual)
vim packages/business-logic/new-entity/custom-logic.function.ts
```

### 2. Don't Edit Generated Files
If you need custom logic:
- ✅ Create new files in business-logic/
- ✅ Extend hooks with custom composition
- ✅ Add middleware for cross-cutting concerns
- ❌ Don't edit @generated files (they'll be overwritten)

### 3. Commit Generated Files
Include generated files in git:
```bash
git add packages/types/
git add packages/resources/repositories/
git add packages/execution/routes/
git commit -m "feat: add pattern entity"
```

This ensures:
- No build step needed to read code
- Code reviews see full changes
- CI validates generated files match entities

### 4. Regenerate After Entity Changes
Always regenerate after changing entities:
```bash
# Changed pattern.entity.ts?
flusk g pattern.entity.ts

# This updates:
# - Types
# - Repository
# - Migration
# - Routes
# - Plugin
```

## Performance Tips

### Parallel Generation

Generate multiple entities in parallel:
```bash
# Use GNU parallel
ls packages/entities/src/*.entity.ts | parallel -j4 flusk g {}

# Or shell background jobs
flusk g llm-call.entity.ts &
flusk g pattern.entity.ts &
flusk g conversion.entity.ts &
wait
```

### Cache Build Artifacts

The CLI is fast because it:
- ✅ Parses TypeBox schemas once
- ✅ Generates files in memory
- ✅ Writes only changed files
- ✅ Uses template caching

Typical generation time: **<2 seconds per entity**

## Summary

**Key Commands:**
- `flusk g entity.entity.ts` - Generate all layers
- `flusk g --all` - Generate from all entities
- `flusk g --types-only` - Types only (fast)
- `pnpm regenerate` - Regenerate everything
- `pnpm check:generated` - CI validation

**What to Know:**
- Entity schemas are SOURCE OF TRUTH
- CLI generates types, repositories, migrations, routes, plugins
- Business logic and hooks are manual (stubs generated)
- All files <100 lines (automatically split)
- @generated headers mark auto-generated files
- Regeneration is safe (manual code preserved)
